**2. Fix CLICK_EVENT quirk (LOOP-06) in `src/bridge/even-bridge.ts`:**
The SDK's `OsEventTypeList.fromJson()` normalizes the numeric value `0` to `undefined` during JSON deserialization. Since `CLICK_EVENT = 0`, tap events have `eventType === undefined` on real hardware.

Current code at line ~76 does `if (eventType === undefined) return;` which drops all tap events on real hardware.

Replace the event handling logic in the `onEvenHubEvent` callback. The current pattern is:
```
if (eventType === undefined) return;
switch (eventType) { case CLICK_EVENT: ... }
```

Change to:
```typescript
// Handle CLICK_EVENT quirk: SDK fromJson normalizes 0 to undefined
if (eventType === OsEventTypeList.CLICK_EVENT || eventType === undefined) {
  bus.emit('gesture:tap', { timestamp: now });
} else {
  switch (eventType) {
    case OsEventTypeList.DOUBLE_CLICK_EVENT:
      bus.emit('gesture:double-tap', { timestamp: now });
      break;
    case OsEventTypeList.SCROLL_TOP_EVENT:
      bus.emit('gesture:scroll-up', { timestamp: now });
      break;
    case OsEventTypeList.SCROLL_BOTTOM_EVENT:
      bus.emit('gesture:scroll-down', { timestamp: now });
      break;
  }
}
```

Remove the early `if (eventType === undefined) return;` guard that currently precedes the switch statement. The `undefined` case now maps to tap instead of being dropped.

Note: The audio frame check (`if (event.audioEvent)`) must remain BEFORE the gesture event handling and must NOT be affected by this change.

<task type="auto">
  <name>Task 2: Create VoiceLoopController module with tests</name>
  <files>src/voice-loop-controller.ts, src/__tests__/voice-loop-controller.test.ts</files>
  <action>
Create a new `src/voice-loop-controller.ts` module following the existing factory+interface pattern (same as gesture-handler.ts, display-controller.ts).

**Interface:**
```typescript
export interface VoiceLoopController {
  destroy(): void;
}
```

**Factory: `createVoiceLoopController(opts)`**
Parameters:
```typescript
opts: {
  bus: EventBus<AppEventMap>;
  gateway: GatewayClient;
  settings: () => AppSettings;
}
```

The `settings` parameter is a getter function (not a direct reference) so it always returns the current settings value even if the user changes gateway URL mid-session. This follows the anti-pattern guidance from research: "Creating gateway client with settings reference instead of getter."

**Behavior:**
1. Subscribe to `gateway.onChunk()` -- forward each chunk to `bus.emit('gateway:chunk', chunk)`
2. Subscribe to `gateway.onStatusChange()` -- forward each status to `bus.emit('gateway:status', { status })`
3. Subscribe to `bus.on('audio:recording-stop')` -- when recording stops, call `gateway.sendVoiceTurn(settings(), { sessionId, audio: blob, sttProvider: settings().sttProvider })`
4. `destroy()` -- unsubscribe all listeners using the unsubs[] array pattern

**Implementation pattern:**
```typescript
const unsubs: Array<() => void> = [];

unsubs.push(gateway.onChunk((chunk) => {
  bus.emit('gateway:chunk', chunk);
}));

unsubs.push(gateway.onStatusChange((status) => {
  bus.emit('gateway:status', { status });
}));

unsubs.push(bus.on('audio:recording-stop', ({ sessionId, blob }) => {
  const s = settings();
  gateway.sendVoiceTurn(s, {
    sessionId,
    audio: blob,
    sttProvider: s.sttProvider,
  });
}));
```

Import types from: `'./types'` for AppSettings/AppEventMap/VoiceTurnChunk/ConnectionStatus, `'./events'` for EventBus, `'./api/gateway-client'` for GatewayClient.
