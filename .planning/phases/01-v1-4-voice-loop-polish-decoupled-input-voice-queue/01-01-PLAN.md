<objective>
Make the gesture-handler's response_end/error reset logic state-aware so that gateway lifecycle events never abort an active user recording.

Purpose: Currently, response_end and error chunks unconditionally call handleInput('reset'), which means if a user starts recording during streaming (now possible thanks to Quick-19/20 FSM transitions), the response_end for the previous turn aborts their new recording by resetting FSM to idle. The state-aware guard ensures only sent/thinking states are eligible for reset.

Output: Modified gesture-handler.ts with state-aware guard, updated tests proving recording immunity.
</objective>

  if (chunk.type === 'error') {

New code:
```typescript
// Only reset FSM on response_end/error when in sent or thinking states.
// Recording, idle, and menu states are immune — prevents aborting an active
// recording when a previous turn's response completes mid-capture.
unsubs.push(bus.on('gateway:chunk', (chunk) => {
  if (chunk.type === 'error' || chunk.type === 'response_end') {
    if (state === 'sent' || state === 'thinking') {
      handleInput('reset', Date.now());
    }
  } else if (chunk.type === 'response_delta') {
    startWatchdog();
  }
}));
```

The `state` variable is already in the closure scope (line 56). The guard checks `state === 'sent' || state === 'thinking'` before calling `handleInput('reset')`.

Note: `state === 'thinking'` is currently a defensive guard -- no FSM transition in `gesture-fsm.ts` produces `nextState: 'thinking'`, so the gesture FSM never reaches that state. The guard is included for forward compatibility: if a `sent -> thinking` transition is added later (e.g., on `response_start` chunk), the guard will already handle it correctly. The display-controller uses a separate `IconState` type for the `thinking` icon, which is unrelated to this FSM state.

This means:
- In `sent` or `thinking`: reset fires (previous turn lifecycle ends, return to idle)
- In `recording`: reset is skipped (user is actively recording a new turn)
- In `idle`: reset is skipped (no-op, already idle)
- In `menu`: reset is skipped (user is browsing menu, irrelevant gateway event)

Do NOT change any other part of the handler. Do NOT modify the FSM transition table (already correct). Do NOT touch the watchdog logic (still works correctly -- watchdog only fires in TRANSIENT_STATES which includes sent/thinking, and after reset the state is idle so watchdog clears).
