# Phase 18.5: Hub Integration Wiring - Research

**Researched:** 2026-03-01
**Domain:** Hub-side integration wiring (event bus emission, IDB handle propagation, health display)
**Confidence:** HIGH

## Summary

Phase 18.5 closes three integration gaps and two broken E2E flows on the hub side, all identified by the v1.3 milestone audit. The gaps are purely wiring issues -- all the underlying modules (hub-error-presenter, drift-reconciler, sync-monitor, health-indicator, storage-health) are fully implemented and tested. The work is confined to a single file (`src/hub-main.ts`) with targeted changes to three areas: (1) emit `persistence:error` events to `hubBus` in hub error paths so the hub error presenter activates, (2) recreate `driftReconciler` and `syncMonitor` with the new IDB handle after `reopenDB()` matching the glasses-side 16.5-01 pattern, and (3) emit `persistence:health` to `hubBus` after `getQuota()` and add `computeStorageHealth()` to `refreshHealthDisplay()` for the storage health dot.

No new modules, no new dependencies, no new types, no HTML/CSS changes. The glasses-side implementation (Phase 16.5-01) provides a verified reference pattern for the reopenDB handle propagation. The hub error presenter (Phase 18-02) is already wired structurally -- it just needs events emitted to `hubBus`.

**Primary recommendation:** Make all three fixes in `src/hub-main.ts` in a single plan. The changes are independent within the file but small enough to bundle. Follow the glasses-side reopenDB pattern exactly for Issue 2, and use `hubBus.emit()` calls alongside existing `addLog()`/`showToast()` calls for Issues 1 and 3.

### Pattern 1: hubBus.emit() Alongside Direct Calls (Issue 1)

**What:** Add `hubBus.emit('persistence:error', ...)` calls alongside existing `addLog()`/`showToast()` calls in hub error paths. The hub error presenter subscribes to these events and provides structured error UX (user-friendly message mapping, 5000ms toast duration, non-recoverable banner).

**When to use:** Every hub-side error path that should activate the error presenter.

**Three error paths to wire:**

```typescript
// Path A: IDB unexpected close (hub-main.ts setOnUnexpectedClose callback, ~line 1103)
// CURRENT:
console.error('[hub] Database connection unexpectedly closed');
addLog('error', 'Database connection unexpectedly closed');
// ADD:
hubBus.emit('persistence:error', {
  type: 'database-closed',
  recoverable: true,
  message: 'Database connection unexpectedly closed',
});

// Path B: IDB reopen failure (hub-main.ts reopenDB().catch(), ~line 1119)
// CURRENT:
addLog('error', 'Database reopen failed -- restart required');
showToast('Database error -- please restart the app');
// REPLACE showToast with hubBus.emit (presenter handles display):
hubBus.emit('persistence:error', {
  type: 'database-closed',
  recoverable: false,
  message: 'Failed to reopen database after max retries',
});

// Path C: Storage eviction detected (hub-main.ts initPersistence(), ~line 1043-1046)
// CURRENT:
console.warn('[hub] Storage eviction detected');
addLog('error', 'Storage eviction detected -- previous conversations were lost');
showToast('Previous data was cleared by system');
// REPLACE showToast with hubBus.emit:
hubBus.emit('persistence:error', {
  type: 'database-closed',
  recoverable: true,
  message: 'Storage eviction detected',
});
```

**Key decision: Replace or supplement showToast?**

The hub error presenter's `showToast` callback IS the hub's `showToast` function (wired at line 968-973). If we emit `persistence:error` to `hubBus`, the presenter will call `showToast` with its own user-friendly message and 5000ms duration. Therefore, **replace** the direct `showToast()` calls with `hubBus.emit()` calls -- do NOT call both, or the user sees a double toast.

For the reopen failure case (Path B), the presenter maps `database-closed` with `recoverable: false` to `showBanner('Data connection lost', 'err')` -- a sticky banner, not a toast. This is better UX than the current 2500ms toast that disappears.

Keep `addLog()` calls -- they go to the diagnostics log, which is separate from user-facing notifications.
Keep `console.error()` / `console.warn()` calls for developer console visibility.

### Pattern 2: Recreate IDB-Dependent Modules After reopenDB (Issue 2)

**What:** After `reopenDB().then((newDb) => ...)`, destroy and recreate `driftReconciler` and `syncMonitor` with the new store, matching the glasses-side pattern from Phase 16.5-01.

**Reference implementation:** `src/glasses-main.ts` lines 136-198 (the reopenDB success handler).

**Hub-side adaptation:**

```typescript
reopenDB().then((newDb) => {
  // 1. Recreate store (already done)
  const newConversationStore = createConversationStore(newDb);
  hubConversationStore = newConversationStore;

  // 2. Recreate driftReconciler with new store
  hubDriftReconciler?.destroy();
  hubDriftReconciler = createDriftReconciler({
    store: newConversationStore,
    onDriftDetected: (info) => {
      console.warn(`[hub] Sync drift: local=${info.localCount} remote=${info.remoteCount}`);
    },
    onReconciled: () => {
      loadLiveConversation();
    },
  });

  // 3. Recreate syncMonitor with new store
  hubSyncMonitor?.destroy();
  hubSyncMonitor = createSyncMonitor({
    bridge: hubSyncBridge!,           // syncBridge is IDB-independent, reuse
    store: newConversationStore,
    origin: 'hub',
    getActiveConversationId: () => sessionManager?.getActiveSessionId() ?? '',
    onHeartbeat: (conversationId, remoteCount) => {
      hubDriftReconciler!.handleHeartbeat(conversationId, remoteCount).catch(() => {});
    },
  });
  hubSyncMonitor.startHeartbeat();

  // 4. Reload and log
  addLog('info', 'Database reconnected -- all modules refreshed');
  loadLiveConversation();
})
```

**Key differences from glasses-side:**
- Hub does NOT recreate `sessionStore` or `autoSave` (hub doesn't have auto-save; hub's `sessionManager` is complex to recreate -- deferred per 16.5-02 decision).
- Hub uses `hubSyncBridge` (module-level) instead of local `syncBridge` variable.
- Hub uses `sessionManager?.getActiveSessionId()` for the getActiveConversationId getter.
- The `onReconciled` callback calls `loadLiveConversation()` (hub pattern) instead of `renderer.destroy()+init()` (glasses pattern).

**Scoping concern:** The reopenDB callback at line 1103 runs inside `initPersistence()`. The `hubDriftReconciler` and `hubSyncMonitor` module-level variables are set in `initHub()` after `initPersistence()` returns (line 954-955). However, `reopenDB().then()` runs asynchronously -- long after `initHub()` completes. By the time the callback fires, `hubDriftReconciler` and `hubSyncMonitor` are set. This is safe. The `hubSyncBridge` is also set at line 953. The `sessionManager` is set at line 952. All module-level references are available.

### Pattern 3: Emit persistence:health to hubBus + Add to refreshHealthDisplay (Issue 3)

**What:** Two changes:
1. In `initPersistence()`, after `getQuota()`, emit `persistence:health` to `hubBus` so the existing subscriber (line 979) fires and updates the storage dot.
2. In `refreshHealthDisplay()`, add a `computeStorageHealth()` call that reads from a module-level cached quota snapshot.

**For the emit (boot-time):**

```typescript
// In initPersistence(), after the existing quota threshold checks (~line 1089-1100)
const quota = await storageHealth.getQuota();
if (quota.isAvailable) {
  // ... existing threshold logging ...
  // ADD: Emit to hubBus so the persistence:health subscriber updates DOM
  hubBus.emit('persistence:health', quota);
}
```

**Scoping concern:** `hubBus` is a module-level `const` declared at line 51. `initPersistence()` is called from `initHub()` at line 950, and the `hubBus.on('persistence:health', ...)` subscriber is registered at line 979 -- AFTER `initPersistence()` returns. This means the emit at boot time would fire BEFORE the subscriber is registered.

**Solution:** Move the `hubBus.emit('persistence:health', quota)` call out of `initPersistence()` and into `initHub()`, after the subscriber registration at line 979. Or, store the quota data and emit it after subscriber registration.

Better approach: Have `initPersistence()` return the quota data, and emit it in `initHub()` after the subscriber is registered. Alternatively, since the subscriber at line 979 is registered in `initHub()` at line 979 and `initPersistence()` is awaited at line 950, the subscriber is NOT registered when `initPersistence()` runs. Therefore, emit the quota data in `initHub()` after line 983 (after the subscriber is registered).

**Implementation approach:**
1. Return the `quota` from `initPersistence()` in its return type (or a separate field).
2. In `initHub()`, after the persistence:health subscriber registration (line 979-983), emit: `if (persistence.quota) hubBus.emit('persistence:health', persistence.quota);`

**For refreshHealthDisplay (ongoing):**

The refreshHealthDisplay function is called when navigating to the health page. Currently it updates sync status but not storage status. To update storage status on page visits, we need cached quota data.

**Two options:**
- Option A: Cache quota at module level, call `computeStorageHealth()` in `refreshHealthDisplay()` using cached data.
- Option B: Re-read quota from `navigator.storage.estimate()` in `refreshHealthDisplay()` (makes it async).

**Recommendation: Option A (cache).** `refreshHealthDisplay()` is currently synchronous. Making it async would require changing all call sites. The quota data from boot is sufficient for the health page -- storage usage doesn't change rapidly enough to warrant real-time re-reads within a single tab session. Store the `StorageHealth` result at module level and use it in `refreshHealthDisplay()`.

```typescript
// Module-level
let cachedQuota: { usagePercent: number; usageBytes: number; quotaBytes: number; isPersisted: boolean } | null = null;

// In initHub(), after persistence:health subscriber:
if (persistence.quota?.isAvailable) {
  cachedQuota = persistence.quota;
  hubBus.emit('persistence:health', persistence.quota);
}

// In refreshHealthDisplay():
if (cachedQuota) {
  const storageSnap = computeStorageHealth(cachedQuota.usagePercent, cachedQuota.usageBytes, cachedQuota.quotaBytes, cachedQuota.isPersisted);
  setHealthDot('hStorageDot', storageSnap.dot);
  $('hStorageStatus').textContent = storageSnap.label;
}
```

### Anti-Patterns to Avoid

- **Double notification:** Do NOT call both `showToast()` directly AND `hubBus.emit('persistence:error')` for the same error. The presenter calls `showToast()` via its callback -- emitting twice produces two overlapping toasts.
- **Recreating sessionManager in reopenDB:** The 16.5-02 plan explicitly decided against this due to complexity. Do NOT attempt it.
- **Async refreshHealthDisplay:** Do NOT make `refreshHealthDisplay()` async to call `getQuota()`. Use cached data instead.

## Common Pitfalls

### Pitfall 1: Subscriber Registration Timing (persistence:health)

**What goes wrong:** Emitting `persistence:health` inside `initPersistence()` fires before the `hubBus.on('persistence:health', ...)` subscriber in `initHub()` is registered.
**Why it happens:** `initPersistence()` is called at line 950 (`await initPersistence()`), but the subscriber is registered at line 979 -- after `initPersistence()` returns.
**How to avoid:** Return quota data from `initPersistence()` and emit it in `initHub()` after subscriber registration. Or, emit it directly in `initHub()` after line 983.
**Warning signs:** Storage health dot shows "Unknown" / off despite quota being available. No errors in console.

### Pitfall 2: Double Toast on Error

**What goes wrong:** User sees two rapid-fire toasts for the same error.
**Why it happens:** Both direct `showToast()` call AND `hubBus.emit('persistence:error')` trigger a toast -- the latter via the hub error presenter's subscription.
**How to avoid:** Replace direct `showToast()` calls with `hubBus.emit()` calls where the error presenter should handle display. Keep `addLog()` for diagnostics.
**Warning signs:** Two toast animations overlapping or one immediately replacing the other.

### Pitfall 3: hubSyncBridge Null in reopenDB Handler

**What goes wrong:** `hubSyncBridge` is null when `createSyncMonitor()` is called in the reopenDB handler.
**Why it happens:** If `initPersistence()` throws before creating `syncBridge` (impossible in current flow, but defensive).
**How to avoid:** Guard `hubSyncBridge` with null check before recreating syncMonitor. If null, skip syncMonitor recreation (degrade gracefully -- drift detection is secondary to store recovery).
**Warning signs:** TypeError: Cannot read properties of null.

### Pitfall 4: Stale Module-Level Variables in Closures

**What goes wrong:** The `onHeartbeat` callback in the recreated syncMonitor captures `hubDriftReconciler` at creation time. If `hubDriftReconciler` is later reassigned again (second reopenDB), the old syncMonitor's callback points to a destroyed reconciler.
**Why it happens:** JavaScript closure captures the variable value, not the reference itself when using `hubDriftReconciler!` in the callback body.
**How to avoid:** Because `hubDriftReconciler` is a module-level `let` variable and the callback uses `hubDriftReconciler!` (reading the variable each time), not a local const, this is actually safe. The callback reads the current value of the module-level variable at invocation time, not at creation time. This is the same pattern used in glasses-main.ts.
**Warning signs:** None expected -- this is safe with module-level `let` variables.

### Pitfall 5: Forgetting loadLiveConversation After Sync Module Recreation

**What goes wrong:** After reopenDB recreates driftReconciler + syncMonitor, conversation data is still stale in the UI.
**Why it happens:** The existing code already calls `loadLiveConversation()` after reopenDB -- but it's called before the sync modules are recreated. Moving it after sync module recreation ensures the freshest state.
**How to avoid:** Call `loadLiveConversation()` as the last step in the reopenDB success handler.
**Warning signs:** UI shows stale data until manual page navigation.

## Open Questions

### 1. Should hub save failures also emit to hubBus?

**What we know:** Hub-side save failures in `handleHubChunk` (line 858-860) and `handleTextSubmit` (line 924-926) currently use `console.error() + showToast()`. These are persistence errors that the hub error presenter could handle with better UX (user-friendly messages, 5000ms duration).

**What's unclear:** Whether adding `hubBus.emit('persistence:error', { type: 'write-failed', recoverable: true })` to these paths is in scope for Phase 18.5 or should be deferred.

**Recommendation:** Include these two save-failure paths in the hubBus emission wiring. They are the same category as the IDB close and eviction paths (hub persistence errors that should go through the presenter). The audit explicitly called out "Hub-side error paths call showToast() directly instead" as the root cause of Issue 1. The phase description says "Wire hubBus.emit('persistence:error') in hub error paths (IDB close, save failure, eviction)." Save failures are explicitly listed.

### 2. Should initPersistence return type change?

**What we know:** `initPersistence()` currently returns `{ sessionManager, sessionStore, syncBridge, conversationStore, syncMonitor, driftReconciler } | null`. To pass quota data back to `initHub()`, we need to add `quota` to this return type.

**What's unclear:** Whether this is the cleanest approach versus alternatives (module-level variable set inside initPersistence).

**Recommendation:** Add `quota: StorageHealth | null` to the return type. This is cleaner than a side-effecting module-level write inside initPersistence, and follows the existing pattern where initPersistence returns all initialized resources.
