#### Issue 1: Hub Error Presenter Dead Code (HIGH)

`createHubErrorPresenter({ bus: hubBus, showToast, showBanner, hideBanner })` is wired at hub-main.ts line 968. The presenter subscribes to `persistence:error` and `fsm:watchdog-reset` on `hubBus`. However, **nothing in the entire codebase ever calls `hubBus.emit(...)`**.

Hub-side error paths use direct function calls instead:
- IDB close: `addLog('error', ...) + showToast('...')` (lines 1104-1105, 1120-1121)
- Save failure: `console.error() + showToast('Message may not be saved')` (lines 858-860, 924-926)
- Eviction: `addLog('error', ...) + showToast('Previous data was cleared')` (lines 993-995)

**Consequences:**
- Hub error presenter's user-friendly message mapping is never used
- Non-recoverable errors never show as sticky banners (only as 2500ms toasts)
- The 5000ms extended toast duration configured in the presenter is never applied
- `fsm:watchdog-reset` fires only in glasses context (gesture-handler runs on glasses); no bridge relays it to hub

**Fix needed:** Emit `persistence:error` to `hubBus` in the hub onclose handler and other error paths, OR wire the hub error paths directly through the presenter.

#### Issue 2: Hub Stale Store After reopenDB (MEDIUM)

Same as previous audit. After hub-side `reopenDB().then((newDb) => { ... })`, only `hubConversationStore` is recreated (lines 1110-1113). The `hubDriftReconciler` and `hubSyncMonitor` created in `initPersistence()` capture the original `conversationStore` in their closures and continue calling `countMessages()` against the stale closed handle.

Glasses-side handles this correctly (Phase 16.5-01: destroys and recreates all 5 modules).

**Impact:** After IDB unexpected close + reopen on hub, drift detection and heartbeat become non-functional until full restart. Rare trigger scenario mitigates severity.

#### Issue 3: Hub Storage Health Dot Never Updates (MEDIUM)

Boot-time quota data is obtained via `storageHealth.getQuota()` at hub-main.ts line 1089 and used for threshold logging via `addLog()`. But it is never emitted to `hubBus`, so the `hubBus.on('persistence:health', ...)` subscriber at line 979 -- which calls `computeStorageHealth()` and `setHealthDot()` -- never fires.

`refreshHealthDisplay()` updates sync dot via `computeSyncHealth()` but has no `computeStorageHealth()` call.

**Result:** Hub health page's Storage row permanently shows "Unknown" / off dot despite having quota data available at boot.
