  <action>
    **menu-controller.ts** -- Create the menu controller module following the project's factory pattern (like createDisplayController, createGestureHandler):

    ```typescript
    export interface MenuController {
      isOpen(): boolean;
      destroy(): void;
    }

    export interface MenuControllerDeps {
      bus: EventBus<AppEventMap>;
      renderer: GlassesRenderer;
      sessionManager: SessionManager;
      getActiveSessionId: () => string;
      onSessionSwitch: (sessionId: string) => Promise<void>;
    }
    ```

    Implementation details:

    1. **State:** `menuState: MenuState | null` (null = menu closed), `autoCloseTimer: ReturnType<typeof setTimeout> | null`

    2. **Auto-close timer (MENU-05):** `const AUTO_CLOSE_MS = 5000;`
       - `resetAutoClose()`: clears existing timer, sets new 5-second timeout that calls `closeMenu()`
       - Reset on every interaction: scroll, select, open
       - On auto-close timeout: call `closeMenu()` which emits `gesture:menu-toggle { active: false }` and emits `menu:close {}` on bus

    3. **Event subscriptions** (collect unsubs for destroy):
       - `bus.on('gesture:menu-toggle', ({ active }) => { if (active) openMenu(); else closeMenu(); })`
       - `bus.on('menu:select', () => { if (!menuState) return; resetAutoClose(); handleSelect(); })`
       - `bus.on('gesture:scroll-up', () => { if (!menuState) return; resetAutoClose(); handleScrollUp(); })`
       - `bus.on('gesture:scroll-down', () => { if (!menuState) return; resetAutoClose(); handleScrollDown(); })`

    4. **openMenu():** Create fresh menu state via `createMenuState()`, render via `renderer.showMenuOverlay(renderMenuText(menuState))`, start auto-close timer.

    5. **closeMenu():** Clear auto-close timer, set menuState to null, call `renderer.restoreConversation()`, emit `gesture:menu-toggle { active: false }` on bus to reset the FSM to idle (the gesture handler's handleInput('reset') path). Actually, use `bus.emit('menu:close', {})` for local notification. For FSM reset, the menu controller should NOT re-emit gesture:menu-toggle (would cause infinite loop since it listens for it). Instead: on auto-close, use a different mechanism. The research recommends injecting a `reset` input to the FSM. Since gesture-handler exposes no public handleInput, the auto-close should emit `gesture:menu-toggle { active: false }` but the menu controller must guard against re-entry: only process `gesture:menu-toggle { active: true }` for opening, and ignore `{ active: false }` (it handles closing internally). So:
       - `bus.on('gesture:menu-toggle', ({ active }) => { if (active && !menuState) openMenu(); })` -- only opens
       - `closeMenu()` is called internally (by auto-close, by command execution, by double-tap close)
       - For double-tap close: the FSM emits `TOGGLE_MENU` which triggers `gesture:menu-toggle { active: false }` (since FSM goes idle). But menu controller ignores active=false. Instead, detect double-tap close via: FSM transitions menu->idle on double-tap with TOGGLE_MENU action. Gesture handler emits `gesture:menu-toggle { active: false }`. Menu controller should listen for this to close: `if (!active && menuState) closeMenu()`. This is safe because closeMenu sets menuState=null before any bus emission, preventing re-entry.
       - Revised subscription: `bus.on('gesture:menu-toggle', ({ active }) => { if (active) openMenu(); else if (menuState) closeMenu(); })`
       - `closeMenu()` sets `menuState = null` FIRST, then clears timer, then calls `renderer.restoreConversation()`

    6. **handleScrollUp/Down():** Call `menuScrollUp(menuState!)` / `menuScrollDown(menuState!)`, update menuState, re-render via `renderer.showMenuOverlay(renderMenuText(menuState))`.

    7. **handleSelect():** Call `menuSelect(menuState!)`.
       - If `result.action === 'confirm'`: update menuState to newState, re-render (shows confirmation dialog)
       - If `result.action === 'execute'`: call `executeCommand(result.command!)`, then `closeMenu()`
       - Note: `menuCancelConfirm` is triggered by scroll during confirmation (menuScrollUp/Down return unchanged state when confirming, so this is handled naturally) OR by double-tap (which closes menu entirely). For explicit cancel: if user scrolls during confirmation, the scrollUp/Down functions already return unchanged state. To allow cancel-by-scroll, modify: if `menuState.confirmingIndex !== null` and user scrolls, call `menuCancelConfirm(menuState)` instead of scrollUp/Down, update state, re-render. This gives users a way to back out of confirmation.

    8. **executeCommand(command: MenuCommand):** Async, fire-and-forget (catch errors, show feedback via renderer.showError if needed):
       - `'new'`: `sessionManager.createSession()` then `onSessionSwitch(newSession.id)`
       - `'switch'`: `sessionManager.loadSessions()`, find next session after current (cycle), if only 1 session show feedback "No other sessions" via brief `renderer.showMenuOverlay('No other sessions')` then after 1 second restore. If multiple, call `onSessionSwitch(nextSessionId)` and `sessionManager.switchSession(nextSessionId)`
       - `'rename'`: Auto-rename using `generateConversationName` pattern. Get most recent user message from the current conversation (via store.getMessages), generate name, call `sessionManager.renameSession(activeId, name)`. If no messages, use timestamp-based name like "Session " + new Date().toLocaleTimeString(). Show brief confirmation text.
       - `'reset'`: Delete all messages from current conversation (call `store.getMessages(id)` then for each message... actually, there's no deleteMessages method on ConversationStore. Instead, delete the conversation and create a new one in the same session slot: `sessionManager.deleteSession(activeId)` then `sessionManager.createSession()` then `onSessionSwitch(newSession.id)`. This effectively "resets" by starting fresh. The old session's messages are cascade-deleted.
       - `'delete'`: `sessionManager.deleteSession(activeId)`, then load remaining sessions. If any exist, switch to first. If none, create new session and switch to it.

       Import `generateConversationName` from `../persistence/conversation-store` for the rename command.

    9. **destroy():** Clear auto-close timer, unsubscribe all bus listeners.

    10. **isOpen():** Return `menuState !== null`.

    **glasses-main.ts** -- Wire menu controller into the boot sequence:

       Note: SessionManager requires sessionStore. If persistence is unavailable (sessionStore is null), skip menu controller creation. Menu commands all require persistence.


    Important: The `createSessionManager` import already exists in sessions.ts. The SessionManager is created here specifically for the menu controller on glasses side. The hub already creates its own SessionManager in hub-main.ts.

Since ConversationStore has getMessages, and the menu controller needs it for rename, add an optional `store: ConversationStore | null` to MenuControllerDeps. Use it only for rename's "get latest message" lookup.
