<feature>
  <name>IntegrityChecker: boot-time data integrity verification</name>
  <files>src/persistence/integrity-checker.ts, src/persistence/__tests__/integrity-checker.test.ts</files>
  <behavior>
    **IntegrityReport interface:**
    - orphanedMessageIds: string[] -- message IDs with conversationId pointing to non-existent conversation
    - danglingPointer: boolean -- localStorage active conversation ID points to deleted conversation
    - sentinelPresent: boolean -- sentinel record exists in conversations store
    - conversationCount: number -- total conversations (excluding sentinel)
    - messageCount: number -- total messages
    - checkedAt: number -- Date.now() timestamp

    **check() behavior (RES-01):**
    - Opens a single readonly IDB transaction over ['conversations', 'messages']
    - Loads all conversation keys into a Set via getAllKeys()
    - Detects sentinel by checking for SENTINEL_ID in keys
    - Scans all messages via cursor to find orphans (conversationId not in convIds Set)
    - Reads localStorage 'openclaw-active-conversation' and checks if it exists in convIds
    - Returns IntegrityReport
    - Under 50 lines of core logic, under 50ms for typical sizes

    Test cases:
    - Empty database: 0 orphans, 0 conversations, 0 messages, no dangling pointer, sentinel not present
    - Database with conversations and messages: 0 orphans, correct counts, sentinel not present
    - Database with orphaned messages: orphan IDs returned correctly
    - Database with sentinel: sentinelPresent = true, sentinel not counted in conversationCount
    - Dangling localStorage pointer: danglingPointer = true
    - Valid localStorage pointer: danglingPointer = false

    **writeSentinel() behavior (RES-04):**
    - Writes a ConversationRecord with id = SENTINEL_ID to conversations store
    - Record has name = '__sentinel__', current timestamp for createdAt/updatedAt
    - Idempotent (put, not add -- overwrites if exists)

    Test cases:
    - writeSentinel creates sentinel record
    - subsequent check() reports sentinelPresent = true
    - writeSentinel is idempotent (calling twice doesn't error)

    **cleanupOrphans(ids) behavior (RES-05):**
    - Takes list of suspected orphan message IDs
    - Re-verifies each by opening a NEW readonly transaction to check if conversationId still missing
    - Only deletes confirmed orphans in a readwrite transaction on messages store
    - Returns count of deleted messages
    - Empty ids array returns 0 immediately

    Test cases:
    - cleanupOrphans([]) returns 0, no IDB operations
    - cleanupOrphans with valid orphan IDs: deletes them, returns count
    - cleanupOrphans with IDs that gained a conversation since detection: skips them (re-verification)
    - cleanupOrphans with mix of valid and stale orphans: only deletes confirmed ones
  </behavior>
  <implementation>
    Use the factory/closure pattern matching conversation-store.ts and session-store.ts:
    ```
    export function createIntegrityChecker(db: IDBDatabase): IntegrityChecker
    ```

    Import SENTINEL_ID from conversation-store.ts.

    Use IDB callback pattern (tx.oncomplete, req.onsuccess) throughout -- never async/await within transactions. This is critical for Safari/WebKit compatibility (see 14-RESEARCH.md Pattern 3).

    For check(): wrap entire function in `new Promise((resolve, reject) => { ... })`. Use getAllKeys() for conversations (fast, no value deserialization). Use openCursor() for messages (need to read conversationId from value). Handle localStorage read in try/catch (may be unavailable).

    For cleanupOrphans(): Two-phase approach:
    1. Re-verify: open readonly tx, getAllKeys on conversations, get each suspected orphan from messages, confirm conversationId still missing
    2. Delete: open readwrite tx on messages, delete confirmed orphan IDs
    Return count of deleted.

    For writeSentinel(): Simple put() on conversations store with sentinel-shaped ConversationRecord.

    Zero new runtime dependencies. Use fake-indexeddb/auto in tests (already in devDependencies).
  </implementation>
</feature>
