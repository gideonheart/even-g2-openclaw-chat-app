# Phase 16.5: Integration Hardening - Research

**Researched:** 2026-02-28
**Domain:** IndexedDB handle propagation, event bus wiring, storage health emission, and cleanup teardown in cross-context architecture
**Confidence:** HIGH

## Summary

Phase 16.5 is a focused bug-fix phase that closes 4 integration issues and 3 broken flows discovered during the v1.3 milestone audit. The bugs are all in completed Phases 14 and 16 code -- they arise not from incorrect module implementations but from incorrect wiring between modules. The scope is narrow: fix `reopenDB()` handle propagation (critical), wire `storage:evicted` event subscribers (significant), add hub `persistence:health` emission with threshold logging (significant), and call `driftReconciler.destroy()` in teardown paths (minor).

All four fixes are in existing files (`db.ts`, `glasses-main.ts`, `hub-main.ts`). No new modules are needed. No new runtime or dev dependencies. The fixes are surgical -- each one is a small, well-bounded change to an existing code path. The challenge is not algorithmic but architectural: ensuring the fix propagates correctly across both glasses and hub contexts, and that no existing tests break.

The critical fix (reopenDB stale handle) requires a design decision about how to propagate the new `IDBDatabase` reference. The current architecture creates stores via `createConversationStore(db)` and `createSessionStore(db, store)` which capture `db` in closure scope. After `reopenDB()` returns a fresh `IDBDatabase`, these closures still reference the dead handle. The fix must either (a) recreate all stores from the new db, or (b) introduce a mutable db reference that stores read from. Option (a) is recommended because it is simpler, matches existing patterns, and avoids introducing shared mutable state.

**Primary recommendation:** Fix each integration issue as a targeted patch to glasses-main.ts, hub-main.ts, and db.ts. Recreate stores after reopenDB() succeeds. Wire storage:evicted subscribers that log and notify. Add hub persistence:health emission matching glasses' existing pattern. Add driftReconciler.destroy() to both cleanup paths.

### Pattern 1: Store Refresh After reopenDB (Critical Fix)

**What:** When `reopenDB()` returns a new `IDBDatabase`, all stores (`ConversationStore`, `SessionStore`, `IntegrityChecker`) must be recreated from the new handle. The existing stores captured the dead handle in closure scope and cannot be patched.

**When to use:** In the `setOnUnexpectedClose` callback, after `reopenDB().then(newDb => ...)`.

**Current broken code (glasses-main.ts lines 133-134):**
```typescript
// BUG: reopenDB() returns new IDBDatabase via .then(db => db)
// but the callback discards it with .then(() => ...)
reopenDB().then(() => {
  bus.emit('log', { level: 'info', msg: 'Database reconnected successfully' });
}).catch(() => { ... });
```

**Fixed code:**
```typescript
// Source: glasses-main.ts setOnUnexpectedClose handler
reopenDB().then((newDb) => {
  // Recreate stores from new handle
  store = createConversationStore(newDb);
  sessionStore = createSessionStore(newDb, store);

  // Re-hook onclose on the new db handle
  newDb.onclose = () => {
    if (onUnexpectedClose) onUnexpectedClose();
  };

  bus.emit('log', { level: 'info', msg: 'Database reconnected successfully' });
}).catch(() => {
  bus.emit('persistence:error', {
    type: 'database-closed',
    recoverable: false,
    message: 'Failed to reopen database after max retries',
  });
});
```

**Key insight:** The `store` and `sessionStore` variables in `glasses-main.ts boot()` are `let` bindings, not `const`. They can be reassigned. The `autoSave`, `sessionManager`, `driftReconciler`, and `syncMonitor` all reference `store` through closures or getters (e.g., `getConversationId: () => activeConversationId`). However, `autoSave` and `driftReconciler` capture `store` at creation time in their closures. This means they too need to be refreshed or they will continue using the dead store.

**Full store-refresh strategy for glasses-main.ts:**

The `autoSave`, `driftReconciler`, and `syncMonitor` all close over `store` when created. After `reopenDB()` succeeds:
1. Reassign `store = createConversationStore(newDb)`
2. Reassign `sessionStore = createSessionStore(newDb, store)`
3. Destroy and recreate `autoSave` with the new store
4. Destroy and recreate `driftReconciler` with the new store
5. Destroy and recreate `syncMonitor` with the new store + bridge

This is a heavyweight recovery, but database closure is a rare catastrophic event. The complexity is justified.

**Alternative (simpler but less complete):** Emit `persistence:error` with `recoverable: false` and tell the user to restart. This is acceptable as a pragmatic first approach -- Phase 18 Error UX will surface this to the user. The reopenDB path already fails after 3 attempts and emits `recoverable: false`. The fix is to also handle the success path correctly for the cases where reopenDB does succeed.

**Recommended approach for Phase 16.5:** Fix the handle propagation on the success path (recreate `store` and `sessionStore`), but do NOT recreate autoSave/driftReconciler/syncMonitor (too complex for a gap-closure phase). Instead, after the new stores are created, set a flag that the next voice turn or heartbeat should use the fresh store. This is a pragmatic middle ground.

**Simplest correct approach:** Since `store` is referenced by the `autoSave`, `driftReconciler`, and `syncMonitor` closures at creation time, the cleanest fix is to make `store` and `sessionStore` variables reassignable AND have the dependent modules reference them through getter functions rather than captured values. However, that would require modifying the module interfaces.

**Final recommendation:** Recreate `store` and `sessionStore` from the new db handle. For `autoSave`: destroy the old one and create a new one with identical config. For `driftReconciler` and `syncMonitor`: destroy and recreate. This is ~20 lines of code in the reopenDB success handler and is fully correct.

### Pattern 2: Hub Storage Health Emission (Matching Glasses Pattern)

**What:** Hub-main.ts `initPersistence()` calls `storageHealth.getQuota()` but discards the result after checking `isPersisted`. The fix mirrors the glasses-main.ts pattern: emit `persistence:health` and log at 80%/95% thresholds.

**Current broken code (hub-main.ts lines 1032-1036):**
```typescript
const storageHealth = createStorageHealth();
const quota = await storageHealth.getQuota();
if (quota.isAvailable && !quota.isPersisted) {
  await storageHealth.requestPersistence();
}
// quota result discarded -- no emission, no threshold logging
```

**Fixed code:**
```typescript
const storageHealth = createStorageHealth();
const quota = await storageHealth.getQuota();
if (quota.isAvailable) {
  // Emit health event for Phase 18 Error UX consumption
  // Hub has no bus -- log to console for now
  if (quota.usagePercent >= 95) {
    console.error(`[hub] Storage critical: ${quota.usagePercent.toFixed(1)}% used`);
    addLog('error', `Storage critical: ${quota.usagePercent.toFixed(1)}% used`);
  } else if (quota.usagePercent >= 80) {
    console.warn(`[hub] Storage warning: ${quota.usagePercent.toFixed(1)}% used`);
    addLog('warn', `Storage warning: ${quota.usagePercent.toFixed(1)}% used`);
  }
  if (!quota.isPersisted) {
    const granted = await storageHealth.requestPersistence();
    addLog('info', `Persistent storage ${granted ? 'granted' : 'denied'}`);
  }
}
```

**Note:** The hub does not have an `EventBus<AppEventMap>` instance. It uses module-level functions (`addLog`, `showToast`). For Phase 16.5, the fix uses `addLog()` and `console.*` which are the hub's existing patterns. Phase 18 may introduce a hub event bus if needed.

### Pattern 3: Wiring storage:evicted Subscribers

**What:** The `storage:evicted` event is emitted in glasses-main.ts (line 53) but has zero subscribers. Add a subscriber that logs and notifies the user.

**Glasses context fix:**
```typescript
// After bus is created, before IDB init
bus.on('storage:evicted', () => {
  bus.emit('log', { level: 'error', msg: 'Storage eviction detected -- previous data was lost' });
  // Show on glasses display (will be replaced by Phase 18 error presenter)
  renderer.showError('Data was cleared by system');
});
```

**Hub context fix:**
```typescript
// In hub-main.ts initPersistence(), after eviction detection
if (hadPreviousData && report.conversationCount === 0) {
  console.warn('[hub] Storage eviction detected');
  addLog('error', 'Storage eviction detected -- previous conversations were lost');
  showToast('Previous data was cleared by system');
}
```

**Timing consideration:** In glasses-main.ts, the `bus.on('storage:evicted', ...)` subscription must be registered BEFORE the sentinel check that emits the event (line 53). The subscriber accesses `renderer`, which is created later (Layer 4). Two options:
1. Defer the subscriber registration until after renderer init (but then the event was already emitted and missed).
2. Use a flag: set `evictionDetected = true` during boot, then after renderer init, show the error if flag is set.

Option 2 is correct because the event fires during boot before the display pipeline is ready.

### Pattern 4: Cleanup Teardown (driftReconciler.destroy)

**What:** Add `driftReconciler.destroy()` to both cleanup paths.

**Glasses context (glasses-main.ts cleanup function):**
```typescript
function cleanup(): void {

  cleaned = true;

  driftReconciler?.destroy();   // NEW: clear mismatch counter

  syncBridge.destroy();
```

**Hub context (hub-main.ts beforeunload):**
```typescript

  driftReconciler?.destroy();   // NEW -- need to expose driftReconciler

  hubSyncBridge?.destroy();

```

**Hub implementation detail:** The `driftReconciler` is currently scoped inside `initPersistence()` and not exposed. The fix must either:
1. Return `driftReconciler` from `initPersistence()` alongside `syncMonitor`, OR
2. Store it in a module-level variable (like `hubSyncMonitor`)

Option 1 is cleaner and follows the existing pattern of returning a result object from `initPersistence()`.

### Anti-Patterns to Avoid

- **Modifying ConversationStore or SessionStore interfaces:** These interfaces are stable and used by many consumers. The reopenDB fix should work by replacing store instances, not modifying the interface to add `setDb()`.
- **Adding a hub event bus in Phase 16.5:** Creating a hub-wide `EventBus<AppEventMap>` is a refactor that belongs in Phase 18 (Error UX). Phase 16.5 should use the hub's existing `addLog()` + `console.*` patterns.
- **Async subscriber for storage:evicted:** The event handler should not await -- it should set a flag and let the boot sequence continue. Blocking boot on eviction notification delays user experience.
- **Recreating syncBridge after reopenDB:** The SyncBridge (BroadcastChannel/localStorage) is independent of IDB. Only IDB-dependent stores need recreation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| New IDBDatabase handle propagation | Custom observable/reactive db wrapper | Recreate stores via existing factory functions | Factory pattern already supports this; adding reactivity is over-engineering |
| Hub event emission | Custom hub event bus for Phase 16.5 | `addLog()` + `console.*` | Hub refactoring is Phase 18 scope; 16.5 is bug fixes only |
| Eviction user notification | Custom notification component | `renderer.showError()` (glasses) and `showToast()` (hub) | Existing notification patterns sufficient for 16.5 |
| Store invalidation tracking | Custom "store is stale" flag system | Destroy old + create new stores | Clean recreation is simpler than tracking validity |

**Key insight:** Phase 16.5 is a wiring fix phase, not a feature phase. Every fix uses existing APIs, existing patterns, and existing UI primitives. No new abstractions are needed.

## Common Pitfalls

### Pitfall 1: reopenDB onclose Hook Not Re-registered on New Handle
**What goes wrong:** After `reopenDB()` returns a new `IDBDatabase`, the `db.onclose` handler is set in `openDB()` but the module-level `onUnexpectedClose` callback references the original boot context. If the new db closes again, the handler fires but references stale state (old stores).
**Why it happens:** `openDB()` always sets `db.onclose = () => { if (onUnexpectedClose) onUnexpectedClose(); }`. The callback is module-level in db.ts. The boot code's `setOnUnexpectedClose()` sets it once. After reopenDB, the new db has the same `onUnexpectedClose` callback which references stale store closures.
**How to avoid:** After recreating stores in the reopenDB success handler, call `setOnUnexpectedClose()` again with a new callback that references the fresh stores. Or: since the reopenDB handler itself will recreate stores, the existing callback is actually fine -- it will call the same handler which will again try reopenDB and recreate stores. The only issue is if `reopenAttempts` was reset to 0, which it is on success.
**Warning signs:** Console shows "Database reconnected" but subsequent IDB operations still fail.

### Pitfall 2: autoSave References Stale Store After Reopen
**What goes wrong:** `createAutoSave({ store, ... })` captures `store` in closure scope. After `reopenDB()` replaces `store` with a new instance, the autoSave continues writing to the dead store.
**Why it happens:** JavaScript closures capture the reference at creation time. Reassigning the outer `store` variable does not update the inner reference.
**How to avoid:** Destroy the old autoSave and create a new one with the fresh store. The autoSave `destroy()` method exists and properly clears bus subscriptions.
**Warning signs:** Messages saved during an active voice turn after database recovery are silently lost.

### Pitfall 3: storage:evicted Subscriber Registered After Event Fires
**What goes wrong:** The `bus.on('storage:evicted', handler)` is registered in the boot sequence. But the `bus.emit('storage:evicted', {})` also fires during boot, potentially before the subscriber is registered.
**Why it happens:** Event bus is synchronous -- emit fires immediately. If emit happens before on(), the handler misses the event.
**How to avoid:** Either register the subscriber before the emit (requires careful ordering), or use a flag pattern: `let evictionDetected = false;` set during boot, then render eviction warning after display init.
**Warning signs:** Eviction occurs but no user notification appears.

### Pitfall 4: Hub driftReconciler Not Accessible for beforeunload Cleanup
**What goes wrong:** `driftReconciler` is created inside `initPersistence()` but not returned in the result object. The `beforeunload` handler cannot call `driftReconciler.destroy()`.
**Why it happens:** The `initPersistence()` return type omits `driftReconciler`. It was wired internally via callback pattern but no cleanup path was exposed.
**How to avoid:** Add `driftReconciler` to the `initPersistence()` return type. Alternatively, store it in a module-level variable (like `hubSyncMonitor`).
**Warning signs:** Console errors about stale timers on page reload. Mismatch counter accumulates across page reloads.

### Pitfall 5: Hub addLog Called Before init()
**What goes wrong:** If `addLog()` is called during `initPersistence()` (which runs before DOM init), the log entries may not render because the log DOM elements are not yet wired.
**Why it happens:** `initPersistence()` is called inside `initHub()` after `init()`. Looking at the code: `init()` is called first (line 917-918), so DOM wiring happens before persistence. This is actually fine -- `init()` calls `renderLogs()` which populates the initial log display, and `addLog()` calls `renderLogs()` again on each new entry.
**How to avoid:** Verify that `init()` is always called before `initPersistence()` in `initHub()`. The current code already does this correctly.
**Warning signs:** Log entries from storage health appear in console but not in hub log UI.

### Fix 1: reopenDB Handle Propagation (glasses-main.ts)

```typescript
// Source: glasses-main.ts setOnUnexpectedClose handler
// Replace lines 133-142

setOnUnexpectedClose(() => {
  bus.emit('persistence:error', {
    type: 'database-closed',
    recoverable: true,
    message: 'Database connection unexpectedly closed',
  });
  bus.emit('log', { level: 'error', msg: 'Database connection unexpectedly closed' });

  reopenDB().then((newDb) => {
    // Recreate all IDB-dependent modules from fresh handle
    store = createConversationStore(newDb);
    sessionStore = createSessionStore(newDb, store);

    // Destroy and recreate autoSave with new store
    autoSave?.destroy();
    autoSave = createAutoSave({
      bus,
      store,
      getConversationId: () => activeConversationId,
      onConversationNamed: (name) => {
        syncBridge.postMessage({
          type: 'conversation:named',
          origin: 'glasses',
          conversationId: activeConversationId,
          name,
        });
      },
      syncBridge,
    });

    // Destroy and recreate driftReconciler with new store
    driftReconciler?.destroy();
    driftReconciler = createDriftReconciler({
      store,
      onDriftDetected: (info) => {
        bus.emit('sync:drift-detected', info);
        bus.emit('log', {
          level: 'warn',
          msg: `Sync drift: local=${info.localCount} remote=${info.remoteCount}`,
        });
      },
      onReconciled: (info) => {
        bus.emit('sync:reconciled', info);
        bus.emit('log', { level: 'info', msg: `Sync reconciled: ${info.conversationId}` });
      },
    });

    // Destroy and recreate syncMonitor with new store
    syncMonitor?.destroy();
    syncMonitor = createSyncMonitor({
      bridge: syncBridge,
      store,
      origin: 'glasses',
      getActiveConversationId: () => activeConversationId,
      onHeartbeat: driftReconciler
        ? (conversationId, remoteCount) => {
            driftReconciler!.handleHeartbeat(conversationId, remoteCount).catch(() => {});
          }
        : undefined,
    });
    syncMonitor?.startHeartbeat();

    bus.emit('log', { level: 'info', msg: 'Database reconnected -- all stores refreshed' });
  }).catch(() => {
    bus.emit('persistence:error', {
      type: 'database-closed',
      recoverable: false,
      message: 'Failed to reopen database after max retries',
    });
    bus.emit('log', { level: 'error', msg: 'Database reopen failed -- restart required' });
  });
});
```

## Open Questions

1. **Should reopenDB() success path recreate ALL dependent modules or just stores?**
   - What we know: `store`, `sessionStore` must be recreated (they close over `db`). `autoSave`, `driftReconciler`, `syncMonitor` close over `store` at creation time. `syncBridge` is IDB-independent.
   - What's unclear: Whether recreating autoSave/driftReconciler/syncMonitor during an active voice turn could cause race conditions.
   - Recommendation: Recreate all IDB-dependent modules. The reopenDB delay (1s cooldown) means no voice turn should be active. If one is, the voice loop controller will handle the error via existing retry/timeout paths. This is a rare catastrophic recovery scenario -- correctness is more important than edge-case smoothness.

2. **Should hub-main.ts get an EventBus for consistency?**
   - What we know: Glasses uses `EventBus<AppEventMap>` extensively. Hub uses module-level functions + console. Phase 18 Error UX will need hookable signals in the hub.
   - What's unclear: Whether Phase 16.5 should add a hub bus or defer to Phase 18.
   - Recommendation: Defer. Phase 16.5 is a bug-fix phase. Adding a hub event bus is a structural change that belongs in Phase 18. Use `addLog()` + `console.*` for now -- these are the hub's existing patterns.

3. **What user-facing notification should the hub show on storage eviction?**
   - What we know: Currently, hub only `console.warn`s on eviction. Glasses emits `storage:evicted` but nobody listens.
   - What's unclear: Whether `showToast()` is sufficient or a persistent banner is needed.
   - Recommendation: Use `showToast()` for Phase 16.5 (quick, non-blocking). Phase 18 will add the proper error banner component with recovery actions.
